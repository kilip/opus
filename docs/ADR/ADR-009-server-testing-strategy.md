# ADR-009: Server Testing Strategy

**Status:** Accepted
**Date:** 2026-05-17
**Deciders:** Chief Architect
**Context:** Opus Server (`opus/server/`)

---

## 1. Context

Opus Server is a modular monolith structured around Clean Architecture (ADR-001). Multiple ADRs
reference testing utilities — `NoopLogger` (ADR-006), `NoopQueue` and `NoopEventBus` (ADR-008),
mock generation conventions (ADR-006, ADR-008) — but no ADR establishes a unified, authoritative
testing strategy.

Without a formal strategy, teams risk inconsistent test organisation, untestable code paths in the
delivery and adapter layers, fragile integration tests that require external infrastructure, and
ambiguity around coverage expectations.

This ADR establishes the canonical testing strategy for all Go code under `opus/server/`, covering
test package conventions, unit testing, integration testing, HTTP handler testing, shared test
utilities, execution commands, and coverage thresholds.

> **Note for AI agents and automated tooling:** This ADR is the authoritative specification for
> all testing conventions in `opus/server/`. Do not invent testing patterns, package structures,
> or build tags beyond what is defined here.

---

## 2. Decision

Opus Server adopts a **layered testing strategy** with two distinct test categories — unit tests
and integration tests — separated by Go build tags. All mocks are generated using
`go.uber.org/mock`. All tests are co-located with the code they test.

---

### 2.1 Test Categories

| Category | File Suffix | Build Tag | Scope |
|---|---|---|---|
| Unit | `_test.go` | _(none)_ | Single package; all dependencies mocked |
| Integration | `_integration_test.go` | `//go:build integration` | Multiple layers; real database (SQLite in-memory) |

---

### 2.2 Test Package Naming Convention

Two naming styles are permitted, each with a defined purpose:

| Style | Declaration | Use Case |
|---|---|---|
| **Black-box** | `package foo_test` | Testing exported API surface only; preferred for service and handler tests |
| **White-box** | `package foo` | Testing unexported functions or internal state; permitted only when necessary |

**Default:** use `package foo_test` (black-box). White-box testing must be justified in a code
comment at the top of the test file.

```go
// internal/agent/service_test.go — black-box (preferred)
package agent_test

// internal/agent/cursor_test.go — white-box (justified: testing unexported cursor encoding)
// White-box: tests unexported encodeCursor/decodeCursor helpers.
package agent
```

---

### 2.3 Unit Tests

#### 2.3.1 Scope and Placement

- Every exported function and method **must** have at least one unit test.
- Unit test files are co-located with the source file they test.
- Unit tests have **no build tag** and run with `go test ./...`.
- All infrastructure dependencies (repositories, logger, queue, event bus) **must** be mocked or
  replaced with the provided `Noop*` implementations.

#### 2.3.2 Mock Generation — `go.uber.org/mock`

All mocks are generated using `go.uber.org/mock/mockgen`. Mock generation directives are placed
in the source file that defines the interface, not in the test file.

```go
// internal/agent/repository.go
package agent

//go:generate mockgen -destination=mock_repository.go -package=agent . Repository

type Repository interface {
    FindByID(ctx context.Context, id string) (*Agent, error)
    FindAll(ctx context.Context, cursor string, limit int) ([]*Agent, string, error)
    Create(ctx context.Context, agent *Agent) (*Agent, error)
    UpdateStatus(ctx context.Context, id string, status Status) error
    Delete(ctx context.Context, id string) error
}
```

Run all mock generation from `server/`:

```bash
go generate ./...
```

Generated mock files carry the standard generated file header and **must never be edited manually**:

```go
// Code generated by MockGen. DO NOT EDIT.
```

#### 2.3.3 Noop Implementations

For dependencies where call assertions are not required, use the provided `Noop*` implementations
in preference to generated mocks to reduce test verbosity:

| Noop Type | Package | Use When |
|---|---|---|
| `logger.NoopLogger` | `internal/shared/logger` | Log output is irrelevant to the test |
| `queue.NoopQueue` | `internal/shared/queue` | Queue enqueue/dispatch is irrelevant |
| `queue.NoopEventBus` | `internal/shared/queue` | Event publish/subscribe is irrelevant |

Use a generated mock (via `go.uber.org/mock`) only when the test must assert that a specific
method was called with specific arguments.

#### 2.3.4 Table-Driven Tests

All unit tests **must** use the table-driven pattern. Inline single-case tests are not permitted
for exported functions.

```go
// internal/agent/service_test.go
package agent_test

import (
    "context"
    "testing"

    "go.uber.org/mock/gomock"
    "github.com/kilip/opus/server/internal/agent"
    "github.com/kilip/opus/server/internal/shared/logger"
)

func TestService_FindByID(t *testing.T) {
    tests := []struct {
        name      string
        agentID   string
        setupMock func(repo *agent.MockRepository)
        wantErr   error
    }{
        {
            name:    "returns agent when found",
            agentID: "agt_001",
            setupMock: func(repo *agent.MockRepository) {
                repo.EXPECT().
                    FindByID(gomock.Any(), "agt_001").
                    Return(&agent.Agent{ID: "agt_001", Name: "Test"}, nil)
            },
            wantErr: nil,
        },
        {
            name:    "returns ErrNotFound when agent does not exist",
            agentID: "agt_missing",
            setupMock: func(repo *agent.MockRepository) {
                repo.EXPECT().
                    FindByID(gomock.Any(), "agt_missing").
                    Return(nil, agent.ErrNotFound)
            },
            wantErr: agent.ErrNotFound,
        },
    }

    for _, tc := range tests {
        t.Run(tc.name, func(t *testing.T) {
            ctrl := gomock.NewController(t)
            mockRepo := agent.NewMockRepository(ctrl)
            tc.setupMock(mockRepo)

            svc := agent.NewService(mockRepo, agent.Config{}, &logger.NoopLogger{})
            _, err := svc.FindByID(context.Background(), tc.agentID)

            if err != tc.wantErr {
                t.Errorf("expected error %v, got %v", tc.wantErr, err)
            }
        })
    }
}
```

### 2.4 Integration Tests

#### 2.4.1 Scope and Build Tag

Integration tests exercise multiple layers together — typically a service calling a real repository
backed by a real database. They are gated behind the `integration` build tag and run only when
explicitly requested.

Every integration test file **must** begin with the build tag declaration:

```go
//go:build integration

package agent_test
```

#### 2.4.2 File Naming

Integration test files use the `_integration_test.go` suffix:

```
internal/agent/service_integration_test.go
adapter/entgo/agent_repo_integration_test.go
```

#### 2.4.3 Test Database — SQLite In-Memory

Integration tests use a **SQLite in-memory database** to eliminate external infrastructure
requirements. The database is created fresh for each test using the helper defined in
`internal/testutil/`.

```go
//go:build integration

package entgo_test

import (
    "context"
    "testing"

    "github.com/kilip/opus/server/internal/testutil"
    "github.com/kilip/opus/server/adapter/entgo"
)

func TestAgentRepo_FindByID(t *testing.T) {
    client := testutil.NewTestEntClient(t)
    repo := entgo.NewAgentRepo(client)

    // seed
    created, err := repo.Create(context.Background(), &agent.Agent{
        ID:   "agt_001",
        Name: "Test Agent",
    })
    if err != nil {
        t.Fatalf("seed failed: %v", err)
    }

    found, err := repo.FindByID(context.Background(), created.ID)
    if err != nil {
        t.Fatalf("FindByID failed: %v", err)
    }
    if found.ID != created.ID {
        t.Errorf("expected ID %s, got %s", created.ID, found.ID)
    }
}
```

#### 2.4.4 Cleanup and Teardown

The `testutil.NewTestEntClient` helper registers `t.Cleanup` automatically. Tests must not
manually close the client.

### 2.5 HTTP Handler Tests

Fiber handlers are tested using Fiber's built-in `app.Test()` helper. Handler tests are unit tests
(no build tag) that mock the service layer.

```go
// delivery/fiber/handler/agent_handler_test.go
package handler_test

import (
    "encoding/json"
    "io"
    "net/http"
    "net/http/httptest"
    "testing"

    "github.com/gofiber/fiber/v3"
    "go.uber.org/mock/gomock"
    "github.com/kilip/opus/server/delivery/fiber/handler"
    "github.com/kilip/opus/server/internal/agent"
)

func TestAgentHandler_GetAgent(t *testing.T) {
    tests := []struct {
        name           string
        agentID        string
        setupMock      func(svc *agent.MockService)
        wantStatusCode int
    }{
        {
            name:    "returns 200 with agent data",
            agentID: "agt_001",
            setupMock: func(svc *agent.MockService) {
                svc.EXPECT().
                    FindByID(gomock.Any(), "agt_001").
                    Return(&agent.Agent{ID: "agt_001", Name: "Test"}, nil)
            },
            wantStatusCode: http.StatusOK,
        },
        {
            name:    "returns 404 when agent not found",
            agentID: "agt_missing",
            setupMock: func(svc *agent.MockService) {
                svc.EXPECT().
                    FindByID(gomock.Any(), "agt_missing").
                    Return(nil, agent.ErrNotFound)
            },
            wantStatusCode: http.StatusNotFound,
        },
    }

    for _, tc := range tests {
        t.Run(tc.name, func(t *testing.T) {
            ctrl := gomock.NewController(t)
            mockSvc := agent.NewMockService(ctrl)
            tc.setupMock(mockSvc)

            app := fiber.New()
            h := handler.NewAgentHandler(mockSvc)
            app.Get("/agents/:id", h.GetAgent)
                        req := httptest.NewRequest(http.MethodGet, "/agents/"+tc.agentID, nil)
            resp, err := app.Test(req)
            if err != nil {
                t.Fatalf("app.Test failed: %v", err)
            }
            defer resp.Body.Close()

            if resp.StatusCode != tc.wantStatusCode {
                t.Errorf("expected status %d, got %d", tc.wantStatusCode, resp.StatusCode)
            }

            // Assert ADR-004 envelope shape
            body, _ := io.ReadAll(resp.Body)
            var envelope map[string]any
            if err := json.Unmarshal(body, &envelope); err != nil {
                t.Fatalf("response is not valid JSON: %v", err)
            }
            if _, ok := envelope["data"]; !ok {
                t.Error("response envelope missing 'data' key")
            }
            if _, ok := envelope["error"]; !ok {
                t.Error("response envelope missing 'error' key")
            }
        })
    }
}
```

**Rules:**

- Handler tests **must** assert the ADR-004 response envelope shape (`data`, `error`, `meta` keys).
- Handler tests **must not** call real services or repositories.
- The Fiber app instantiated in handler tests uses default configuration (no middleware) unless the
  test explicitly requires middleware behaviour.

### 2.6 Shared Test Utilities — `internal/testutil/`

Shared test helpers are defined in `internal/testutil/`. This package is the only permitted
location for cross-package test utilities.

```
opus/
└── server/
    └── internal/
        └── testutil/
            ├── ent.go        # NewTestEntClient — in-memory SQLite Ent client
            ├── fiber.go      # NewTestFiberApp — preconfigured Fiber app for handler tests
            └── fixtures.go   # Seed helpers for common domain entities
```

```go
// internal/testutil/ent.go
package testutil

import (
    "context"
    "testing"

    "github.com/kilip/opus/server/ent"
    "github.com/kilip/opus/server/adapter/entgo"
    "github.com/kilip/opus/server/internal/config"
)

// NewTestEntClient creates a new in-memory SQLite Ent client for integration tests.
// The client is automatically closed when the test ends via t.Cleanup.
func NewTestEntClient(t *testing.T) *ent.Client {
    t.Helper()
    cfg := config.DatabaseConfig{
        Driver: "sqlite3",
        DSN:    "file:testdb?mode=memory&cache=shared&_fk=1",
    }
    client, err := entgo.NewClient(cfg)
    if err != nil {
        t.Fatalf("testutil: failed to create test ent client: %v", err)
    }
    if err := client.Schema.Create(context.Background()); err != nil {
        t.Fatalf("testutil: failed to run schema migration: %v", err)
    }
    t.Cleanup(func() { _ = client.Close() })
    return client
}
```

**Rules:**

- `internal/testutil/` is **test-only**. It must not be imported by non-test production code.
- Helpers in `testutil/` must call `t.Helper()` as their first statement.
- Fixtures in `testutil/fixtures.go` provide pre-seeded domain entities for common test scenarios.

### 2.7 Test Execution

#### Unit Tests

```bash
# Run all unit tests
go test ./...

# Run with race detector (required in CI)
go test -race ./...

# Run a specific package
go test ./internal/agent/...
```

#### Integration Tests

```bash
# Run all integration tests
go test -tags integration ./...

# Run integration tests with race detector
go test -race -tags integration ./...
```

#### Combined (CI)

```bash
# Full test suite — unit + integration + race detector
go test -race -tags integration ./...
```

### 2.8 Coverage Thresholds

| Category | Threshold | Enforcement |
|---|---|---|
| Unit tests | **80%** per package | CI fails below threshold |
| Integration tests | **70%** per adapter package | CI fails below threshold |

Coverage is measured per package, not globally. Packages that contain only generated code
(`ent/`) and `internal/testutil/` are excluded from coverage measurement.

```bash
# Generate coverage report
go test -race -coverprofile=coverage.out ./...
go tool cover -html=coverage.out -o coverage.html

# Check threshold (example using go-test-coverage or similar)
go test -race -coverprofile=coverage.out ./...
```

> **Note:** Coverage thresholds are enforced in CI via the project's coverage tooling. Local
> development is not blocked by coverage, but engineers are expected to meet the threshold before
> opening a pull request.

## 3. Alternatives Considered

### 3.1 Separate `test/` Directory for Shared Utilities

A top-level `test/` directory (e.g. `server/test/helpers/`) for shared test utilities. Rejected
because:

- Go convention discourages deep test-only package hierarchies at the module root
- `internal/testutil/` is more discoverable and consistent with the feature-based layout from
  ADR-001
- The `internal/` boundary prevents `testutil/` from being imported by external modules
  accidentally

### 3.2 `testify` as Primary Assertion Library

Using `github.com/stretchr/testify` for assertions. Not adopted as a requirement because:

- Go's standard `testing` package is sufficient for the assertion patterns used in Opus
- `testify` is not prohibited — teams may use it where it genuinely reduces verbosity — but it is
  not mandated, avoiding a forced dependency
- `go.uber.org/mock` is mandated for mocking; testify's mock package (`testify/mock`) must not
  be used

### 3.3 External Test Database (PostgreSQL in Docker)

Running integration tests against a real PostgreSQL instance in Docker. Rejected for the default
integration test configuration because:

- Requires Docker to be available in all CI and local development environments
- SQLite in-memory covers all required integration scenarios for the adapter layer
- PostgreSQL-specific behaviour (e.g. `SELECT ... FOR UPDATE SKIP LOCKED`) is tested via
  dedicated CI pipeline stages that provision a PostgreSQL service container, defined separately
  from this ADR

## 4. Consequences

### 4.1 Positive

- **Zero infrastructure for unit tests** — `Noop*` implementations and generated mocks eliminate
  all external dependencies from the default test run
- **Zero infrastructure for integration tests** — SQLite in-memory removes Docker or PostgreSQL
  requirements from the standard integration test suite
- **Consistent mock strategy** — `go.uber.org/mock` everywhere; no mixed use of testify/mock
- **Clear separation** — Build tag `integration` makes it impossible to accidentally run
  integration tests in a unit-test-only context
- **ADR-004 compliance in handler tests** — Mandatory envelope shape assertions prevent handlers
  from silently breaking the response contract
- **Discoverable test utilities** — Single `internal/testutil/` package; no scattered helper files

### 4.2 Negative / Trade-offs

- **SQLite ≠ PostgreSQL exactly** — Integration tests against SQLite will not catch
  PostgreSQL-specific issues (e.g. constraint differences, `SKIP LOCKED` behaviour). A separate
  CI stage with a PostgreSQL service container is required for full coverage of the PostgreSQL
  adapter
- **80% threshold per package** — Small packages with complex setup may find 80% difficult to
  achieve without over-engineering tests; exemptions require a code comment justification and
  per-package exclusion in the CI configuration
- **`go generate` discipline** — Mock files must be regenerated whenever an interface changes;
  stale mocks cause subtle test failures. A CI lint step checking that generated files are
  up-to-date is strongly recommended
