# ADR-003: Opus Dash Frontend Architecture

**Status:** Accepted  
**Date:** 2026-05-17  
**Deciders:** Chief Architect  
**Context:** Opus Dash (`opus/dash/`)

---

## 1. Context

Opus requires a web-based dashboard — **Opus Dash** — to expose its agent lifecycle, vault, and workflow domains to the end user. Opus Dash runs as a **Progressive Web Application (PWA)** served locally alongside the Opus Server.

Because Opus is a local-first, self-hosted system, there is no requirement for server-side rendering (SSR) or static site generation (SSG). Opus Dash communicates exclusively with the Opus Server REST/SSE API over localhost. The PWA layer provides installability, an offline-capable app shell, cached reads, and a mutation queue that synchronises automatically when the Opus Server becomes available again.

This ADR establishes the technology stack, PWA architecture, project structure, and conventions for all frontend code under `opus/dash/`.

---

## 2. Decision

Opus Dash adopts a **feature-based TypeScript PWA** using Vite as the build tool, React as the UI framework, and a curated stack aligned with the feature-based clean architecture established in ADR-001.

### 2.1 Technology Stack

| Concern | Library / Tool | Version |
|---|---|---|
| Build tool | Vite | 6.x |
| UI framework | React | 19.x |
| Language | TypeScript | 5.x |
| Routing | TanStack Router | 1.x |
| Server state | TanStack Query | 5.x |
| UI components | shadcn/ui | latest |
| Styling | Tailwind CSS | 4.x |
| PWA / Service Worker | vite-plugin-pwa | 0.x |
| UI state | React Context + `useState` | — |

#### Rationale

- **Vite** is chosen over Next.js and Remix. Opus Dash is a pure SPA; SSR provides no benefit for a local-first application, and Vite's dev server startup and HMR performance are significantly faster.
- **TanStack Router** is chosen over React Router v7 for its first-class TypeScript support, type-safe route parameters, and seamless integration with TanStack Query's loader pattern.
- **TanStack Query** manages all server state — API responses, cache invalidation, background refetching, and optimistic updates. Its `onMutate` / `onError` / `onSettled` lifecycle hooks integrate cleanly with the offline mutation queue.
- **vite-plugin-pwa** is chosen over Serwist for its zero-configuration Workbox integration and native Vite plugin API. Serwist provides finer-grained control but adds verbosity that is not justified at this stage; it may be introduced in a future ADR if custom service worker logic is required.
- **shadcn/ui** provides accessible, unstyled component primitives composed with Tailwind CSS. Components are copied into the project rather than imported from a package, eliminating upstream dependency risk.
- **Tailwind CSS v4** provides utility-first styling co-located with component markup.
- **React Context + `useState`** is sufficient for UI state (theme, sidebar, modals) at MVP scale. Zustand is deferred until complexity justifies it (YAGNI).

### 2.2 PWA Architecture

#### 2.2.1 Service Worker Strategy

The service worker is generated by `vite-plugin-pwa` using Workbox's `generateSW` mode. It implements a **hybrid offline strategy**:

| Asset / Request Type | Caching Strategy | Rationale |
|---|---|---|
| App shell (HTML, JS, CSS bundles) | `CacheFirst` (precache) | Static assets never change between deploys; serve from cache instantly |
| Static assets (fonts, icons, images) | `CacheFirst` (runtime cache) | Immutable; no benefit to network round-trip |
| `GET /**` | `NetworkFirst` with fallback to cache | Serve fresh data when online; fall back to last cached response when offline |
| `POST / PUT / DELETE /**` | Background sync queue | Mutations are queued in IndexedDB when offline; replayed automatically when the server becomes reachable |

#### 2.2.2 Offline Read Strategy (NetworkFirst + Cache Fallback)

All `GET` API requests use Workbox's `NetworkFirst` strategy with a configurable timeout. If the network request fails or times out, the service worker serves the most recent cached response.

```typescript
// vite.config.ts (relevant excerpt)
import { VitePWA } from 'vite-plugin-pwa';

export default defineConfig({
  plugins: [
    VitePWA({
      registerType: 'autoUpdate',
      workbox: {
        globPatterns: ['**/*.{js,css,html,ico,png,svg,woff2}'],
        runtimeCaching: [
          {
            urlPattern: /^http:\/\/localhost:8080\/api\/.*/,
            handler: 'NetworkFirst',
            options: {
              cacheName: 'opus-api-cache',
              networkTimeoutSeconds: 5,
              expiration: { maxEntries: 200, maxAgeSeconds: 86400 },
              cacheableResponse: { statuses: [0, 200] },
            },
          },
        ],
      },
    }),
  ],
});
```

#### 2.2.3 Offline Write Strategy (Background Sync Queue)

Mutations (`POST`, `PUT`, `DELETE`) that fail due to network unavailability are captured by the service worker and stored in a Workbox `BackgroundSyncPlugin` queue backed by IndexedDB. The queue replays automatically when the service worker detects network restoration via the `sync` event.

```typescript
// vite.config.ts (continued)
runtimeCaching: [
  // ... GET strategy above ...
  {
    urlPattern: /^http:\/\/localhost:8080\/api\/.*/,
    method: 'POST',
    handler: 'NetworkOnly',
    options: {
      backgroundSync: {
        name: 'opus-mutation-queue',
        options: { maxRetentionTime: 24 * 60 }, // 24 hours in minutes
      },
    },
  },
],
```

TanStack Query's `onMutate` callback performs **optimistic updates** in the client-side cache before the mutation is sent. If the network is unavailable and the request is queued, the UI reflects the optimistic state immediately; the service worker replays the mutation when the server is reachable again.

#### 2.2.4 Offline Indicator

A shared `useNetworkStatus()` hook subscribes to the browser's `online` / `offline` events and exposes an `isOnline` boolean. The root layout renders an `OfflineBanner` component when `isOnline` is `false`, informing the user that reads are served from cache and writes are queued.

```typescript
// shared/hooks/useNetworkStatus.ts
import { useEffect, useState } from 'react';

export function useNetworkStatus() {
  const [isOnline, setIsOnline] = useState(navigator.onLine);

  useEffect(() => {
    const handleOnline = () => setIsOnline(true);
    const handleOffline = () => setIsOnline(false);
    window.addEventListener('online', handleOnline);
    window.addEventListener('offline', handleOffline);
    return () => {
      window.removeEventListener('online', handleOnline);
      window.removeEventListener('offline', handleOffline);
    };
  }, []);

  return { isOnline };
}
```

#### 2.2.5 Web App Manifest

The manifest is generated by `vite-plugin-pwa` from the configuration in `vite.config.ts`. It enables installation on desktop and mobile platforms.

```typescript

VitePWA({
```

#### 2.2.6 Service Worker Update Flow

`vite-plugin-pwa` is configured with `registerType: 'autoUpdate'`. When a new service worker is available, it activates automatically after the current session's navigation. A `useServiceWorkerUpdate()` hook exposes `needsRefresh` and `updateServiceWorker()` for cases where the UI should prompt the user to reload.

### 2.4 Layer Responsibilities

| Layer | Path | Responsibility |
|---|---|---|
| **App** | `src/app/` | Entry point, router tree, global provider composition |
| **Routes** | `src/routes/` | Page-level components; TanStack Router file-based route definitions |
| **Features** | `src/features/[feature]/` | Domain-scoped components, hooks, API query definitions, and types |
| **Shared** | `src/shared/` | Cross-feature UI primitives, hooks, API client, PWA utilities, and common types |

---

### 2.5 Feature Module Convention

Each feature module is self-contained. Code within a feature may import from `shared/` but must not import from sibling features. Cross-feature data requirements are resolved at the route level.

**`api.ts` pattern:**

```typescript
// features/agent/api.ts
import { queryOptions } from '@tanstack/react-query';
import { apiClient } from '@/shared/lib/api-client';
import type { AgentRun } from './types';

export const agentKeys = {
  all: ['agents'] as const,
  detail: (id: string) => ['agents', id] as const,
};

export const agentQueries = {
  list: () =>
    queryOptions({
      queryKey: agentKeys.all,
      queryFn: () => apiClient.get<AgentRun[]>('/agents'),
    }),
  detail: (id: string) =>
    queryOptions({
      queryKey: agentKeys.detail(id),
      queryFn: () => apiClient.get<AgentRun>(`/agents/${id}`),
    }),
};
```

**Hook pattern:**

```typescript
// features/agent/hooks/useAgent.ts
import { useQuery } from '@tanstack/react-query';
import { agentQueries } from '../api';

export function useAgent(id: string) {
  return useQuery(agentQueries.detail(id));
}
```

### 2.6 Routing Convention

Routes are defined in `src/routes/` using TanStack Router's file-based convention. The route tree is registered in `src/app/router.ts`.

- `__root.tsx` — root layout route; renders `<Sidebar>`, `<Header>`, `<OfflineBanner>`, and `<Outlet>`
- `index.tsx` — default redirect to `/agent`
- `[feature]/index.tsx` — feature list view
- `[feature]/$id.tsx` — feature detail view with type-safe `$id` param

Route components are thin: they compose feature components and wire TanStack Query loaders. No business logic or direct API calls are made in route files.

```typescript
// routes/agent/$agentId.tsx
import { createFileRoute } from '@tanstack/react-router';
import { agentQueries } from '@/features/agent/api';
import { AgentDetail } from '@/features/agent/components/AgentDetail';

export const Route = createFileRoute('/agent/$agentId')({
  loader: ({ context: { queryClient }, params }) =>
    queryClient.ensureQueryData(agentQueries.detail(params.agentId)),
  component: AgentDetailPage,
});

function AgentDetailPage() {
  const { agentId } = Route.useParams();
  return <AgentDetail agentId={agentId} />;
}
```

### 2.7 API Client

A single configured API client instance is defined in `shared/lib/api-client.ts`. All feature `api.ts` files import from this shared instance. This centralises base URL configuration, authentication header injection, and error handling.

```typescript
// shared/lib/api-client.ts
const BASE_URL = import.meta.env.VITE_API_URL ?? 'http://localhost:8080';

export const apiClient = {
  get: async <T>(path: string): Promise<T> => {
    const res = await fetch(`${BASE_URL}${path}`);
    if (!res.ok) throw new Error(res.statusText);
    return res.json() as Promise<T>;
  },
  post: async <T>(path: string, body: unknown): Promise<T> => {
    const res = await fetch(`${BASE_URL}${path}`, {
      method: 'POST',
      headers: {
        'Content-Type': 'application/json',
      },
      body: JSON.stringify(body),
    });
    if (!res.ok) throw new Error(res.statusText);
    return res.json() as Promise<T>;
  },
};
```

### 2.8 UI State Management

Global UI state (theme, sidebar visibility) is managed with React Context and `useState`. No external state manager is introduced at MVP.

If UI state complexity grows (e.g. cross-feature modal coordination, undo stacks), introduction of Zustand will be addressed in a future ADR.

The Opus Dash feature modules mirror the server domain packages defined in ADR-001:

|---|---|
| `agent` | `agent` |

This 1:1 mapping ensures that developers navigating either codebase can reason about domain boundaries consistently.

## 3. Alternatives Considered

### 3.1 Next.js

Provides SSR, SSG, file-based routing, and a large ecosystem. Rejected because:

- SSR provides no value for a local-first SPA with no public-facing routes
- Introduces Node.js server runtime complexity that duplicates the Opus Go server
- PWA integration requires `next-pwa` or Serwist, both of which add Next.js-specific boilerplate

### 3.2 Remix

Full-stack framework with loader/action pattern. Rejected because:

- Requires a Node.js server process, conflicting with Opus's self-hosted Go server model
- The loader/action pattern is redundant alongside TanStack Query's client-side caching and offline queue

### 3.3 React Router v7

Mature, widely adopted routing library. Not chosen in favour of TanStack Router because:

- TanStack Router provides first-class TypeScript type inference for route params, search params, and loader data without additional boilerplate
- Natural integration with TanStack Query via the `context.queryClient` loader pattern

### 3.4 Serwist

Modern Workbox wrapper with fine-grained service worker control. Deferred (not rejected) because:

- `vite-plugin-pwa`'s built-in Workbox integration covers all required caching and background sync strategies for MVP
- Serwist's additional control is not justified until custom service worker logic is required beyond what Workbox's `generateSW` mode provides

### 3.5 Zustand for Global State

Lightweight global state manager. Deferred (not rejected) because:

- React Context + `useState` is sufficient for MVP-scale UI state
- Will be revisited if cross-feature state coordination becomes necessary

## 4. Consequences

### 4.1 Positive

- **Installable** — The PWA manifest enables installation on desktop (Windows, macOS, Linux) and mobile (iOS, Android), giving Opus Dash a native app feel without a separate Electron or Tauri build
- **Offline reads** — `NetworkFirst` with cache fallback ensures the dashboard remains usable during Opus Server restarts, updates, or brief network interruptions
- **Offline writes** — Background sync queue prevents data loss; mutations are replayed automatically without user intervention
- **Type safety end-to-end** — TanStack Router and TanStack Query provide TypeScript inference from API response to rendered component
- **Feature co-location** — All domain logic for a feature lives in one directory, matching the server-side convention from ADR-001
- **Zero SSR complexity** — Pure SPA; the dashboard is a static bundle served by the Go server

### 4.2 Negative / Trade-offs

- **Background sync browser support** — The Background Sync API is not supported in all browsers (notably Safari as of 2026). Mutations on unsupported browsers fall back to immediate failure with a user-facing error toast; queuing is not available
- **Service worker debugging complexity** — Service workers introduce a caching layer that can obscure bugs during development; the service worker must be disabled in dev mode via `{ devOptions: { enabled: false } }` in `vite-plugin-pwa`
- **Cache invalidation discipline** — `NetworkFirst` with `maxAgeSeconds: 86400` means stale data may be served for up to 24 hours if the server is unreachable; cache TTL must be tuned per endpoint as the API surface matures
- **shadcn/ui maintenance burden** — Vendored components require manual updates when upstream fixes are needed
- **`shared/` discipline** — Components and hooks must be genuinely cross-feature before promotion to `shared/`
